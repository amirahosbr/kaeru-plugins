🐸 Kaeru

Kaeru カエル

Let non-technical Japanese staff safely tweak a live client site — themselves.

A guardrailed way to vibe-code small UI/UX changes on existing client sites — described in Japanese, seen live — while sensitive parts like the CMS, config and logic stay off-limits. The developer approves a pull request before anything ships.

The problem

Every tiny edit lands on the developer

Amirah (Malaysia) maintains client websites — Noah, NRHA and others. Two PMs in Japan, with limited English, constantly need small UI/UX changes: a sentence, an image, a color. Today Amirah does every single one, and the language and timezone gap makes hand-holding slow.

Kaeru moves those small edits to the PMs — safely — while the developer keeps final control by reviewing a pull request before it ships.

How it works

Four steps, in Japanese, git hidden

  1. PM
    /kaeru:start — opens a fresh workspace (a branch, invisibly)
  2. PM
    /kaeru:edit-text · /kaeru:edit-image — makes the change by describing it
  3. PM
    /kaeru:preview — sees the result rendered live and confirms「これでOK?」
  4. PM → Dev
    /kaeru:submit — opens a pull request. The plugin never merges — Amirah reviews the pixels and merges.
Two safety layers: guardrails inside the repo (Japanese rules: text/image/color only, never touch config or logic) and GitHub branch protection. Even if the agent errs, main is protected.
The commands

Eight commands, plain and small

English names, Japanese inside. Most are for the PM; two are for the developer.

/kaeru:startPMBegin a change — pulls the latest and opens a clean workspace. (A branch, invisibly.)
/kaeru:edit-textPMChange wording or copy on the site — just say what to change.
/kaeru:edit-imagePMSwap an image for a new one.
/kaeru:previewPMSee the change rendered live in the browser, with a screenshot, before sending.
/kaeru:submitPMSend it for review — opens a pull request. Never merges; the developer approves.
/kaeru:helpPMExplain an error in plain Japanese and write a memo to forward to the developer.
/kaeru:learnPMAn interactive Japanese tutorial — practice the whole flow, step by step.
/kaeru:setupDevRun once per repo: detect the framework and write the repo map, so edits know where things live.
The core decision

Two kinds of change, two homes

Don't force everything through code. Content lives in the CMS; minor UI/UX lives in Kaeru; and the sensitive core stays off-limits to everyone but the developer.

News / content

→ microCMS

Already Japanese, safe, free. PMs edit news pages directly in the CMS — no Kaeru, no git needed.

Minor UI / UX

→ Kaeru

Text, images and colors baked into the codebase. Vibe-coded through guardrails + preview + PR — this is Kaeru's job.

Sensitive core

→ Off-limits

Auth, config, logic, data, secrets. The agent must never touch these — guardrails and branch protection enforce it.

Built so far

The plugin is live

  • Japanese-first with inline guardrails; framework-neutral via a per-repo map (/kaeru:setup)

  • Marketplace + auto-enable; one-command installers for macOS and Windows

Guiding principle
Prove the boring backbone — guardrails, PR, preview — with two PMs on one pilot repo first. Then scale. Don't build the cloud before the workflow is real.
